# Explore

"만들까 말까?"를 고민하는 공간. 질문과 실시간 조사를 인라인으로 엮어 판단 문서를 만든다.
스펙이 아니라 **결정**이 출력이다.

## 언제 사용

- 아이디어가 있는데 해볼 가치가 있는지 모를 때
- "이런 게 이미 있나?" 궁금할 때
- 시장조사 + 방향 고민을 동시에 하고 싶을 때
- 스타트업 아이디어의 현실성을 진단하고 싶을 때

## 사용하지 말 것

- 이미 만들기로 결정했을 때 → `/ina:think`
- 순수 정보 조사만 필요할 때 → `/ina:research`
- 요구사항이 명확할 때 → `/ina:plan`

## ina 연동

ina 데몬에 의해 실행된 경우:

- Phase 1 시작: `ina_report_progress(in_progress="탐색 인터뷰 중 ({mode} 모드)")`
- Landscape: `ina_report_progress(in_progress="시장/경쟁 조사 중")`
- 판정: `ina_report_progress(in_progress="판단 문서 작성", completed="탐색 완료")`
- NO-GO: `ina_mark_blocked(reason="NO-GO 판정 — 사용자 결정 대기")`

## 전체 흐름

```
Phase 0: 분류 (Startup / Builder 자동 감지 + 확인)
Phase 1: 인터뷰 (모드별 질문 + 인라인 WebSearch)
Phase 2: Landscape (시장/경쟁 병렬 조사)
Phase 3: Premise Challenge (전제 도전)
Phase 4: 판단 문서 저장 (GO / NO-GO / PIVOT)
```

## Phase 0: 분류

사용자 입력을 분석하여 모드를 자동 선택:

| 시그널 | 모드 | 성격 |
|--------|------|------|
| "스타트업", "수익", "SaaS", "B2B", "사업", 고객/매출 관련 | **Startup** | 진단형 — 증거를 요구하고 현실을 직시 |
| "사이드", "배우고 싶어", "해보고 싶어", "취미", "개인 프로젝트" | **Builder** | 탐험형 — 가능성을 열고 영감을 발견 |

감지 결과 확인: "**{mode}** 모드로 탐색합니다. 맞나요?"

**Escape hatch**: 사용자가 "그냥 만들어" / "바로 시작해" 라고 하면:
→ "아직 탐색이 필요한 단계인 것 같아서 explore를 열었습니다. 바로 `/ina:think`로 넘어갈까요?"

## Phase 1: 인터뷰 + 인라인 리서치

**핵심 규칙**: 한 번에 한 질문. 아첨 금지. 입장을 취한다. 모호한 답변에는 반드시 반박한다.

질문과 WebSearch를 자연스럽게 엮는다 — "잠깐, 이 부분은 바로 찾아볼게요."

**Privacy gate** (첫 질문 전에 한 번만):
> "검색을 통해 경쟁사/시장 정보를 실시간으로 조사할 예정입니다. 스텔스 아이디어라면 검색을 건너뛸 수 있습니다. 어떻게 할까요?"

---

### Startup 모드 — 6 Diagnostic Questions

제품 단계 감지:
- **pre-product**: 6개 전부
- **has-users**: Q1 간소화 (관심 → 행동 증거)
- **paying**: Q1·Q5 간소화

**6 Diagnostic Questions** (한 번에 하나, WebSearch 인라인 가능):

1. **절박한 구체성**: "이게 가장 필요한 사람을 한 명 묘사하세요. 이름, 직함, 상황."
   - 모호한 답변 반박: "그건 특정인이 아니라 인구집단입니다. 당신이 직접 아는 사람은?"

2. **현재 대안**: "그 사람이 지금 이 문제를 어떻게 해결하고 있나요? 돈을 내고 있나요?"
   - 즉시 검색: 언급된 대안 제품을 WebSearch로 확인

3. **왜 지금**: "1년 전엔 왜 이 아이디어가 안 됐을까요? 무엇이 바뀌었나요?"
   - 검색: 시장 트렌드, 관련 기술 변화

4. **최소 쐐기**: "이번 주 안에 누군가가 돈을 낼 가장 작은 버전은?"
   - Pushback: "그건 제품 전체입니다. 딱 하나의 기능으로 줄이면?"

5. **경쟁 현실**: "이미 이걸 하고 있는 팀이 있나요?"
   - 즉시 검색: `{아이디어 키워드} startup` / `{아이디어 키워드} alternative`

6. **수익 경로**: "어느 고객이 얼마를 낼까요? 연간 계약인가요, 트랜잭션인가요?"

---

### Builder 모드 — 5 Exploration Questions

발산적 사고를 장려한다. 틀린 방향이 없다. 단, 현실 점검은 한다.

1. **핵심 호기심**: "이걸 만들면서 가장 배우고 싶은 것은? 기술인가요, 도메인인가요?"

2. **선례 탐색**: "비슷한 게 이미 있나요? 없다면 없는 이유가 있을까요?"
   - 즉시 검색: `{아이디어 키워드} open source` / `{아이디어 키워드} side project`

3. **완성의 기준**: "언제 '됐다'고 할 수 있나요? 가장 작은 완성 형태는?"

4. **마찰 예측**: "이걸 완성하지 못한다면 어느 단계에서 막힐 것 같나요?"

5. **영감 발견**: "이와 비슷하지만 다른 방식으로 한 프로젝트를 찾아볼게요."
   - WebSearch: 인접 프로젝트 탐색 + 영감 포인트 제시

---

**최소 질문 수**: 모드에 관계없이 최소 2개는 반드시 질문한다.
사용자가 조기 종료를 원하면: "지금 정보로 Landscape와 판정을 진행할 수 있지만, 판단의 정확도가 낮을 수 있습니다. 계속할까요?"

## Phase 2: Landscape (시장/경쟁 조사)

인터뷰 완료 후, 수집된 맥락으로 병렬 조사를 수행한다.

**Privacy gate에서 검색 거부 시**: 이 Phase를 건너뛴다.

3개 검색 관점을 **하나의 메시지에서 병렬 Agent로** 실행:

```
Agent 1 — Competitive Landscape
  검색: "{핵심 키워드} alternatives 2024"
  검색: "{핵심 키워드} startup"
  찾는 것: 직접 경쟁자, 간접 대안, 포지셔닝 공백

Agent 2 — Market Signal
  검색: "{핵심 키워드} market size"
  검색: "{핵심 키워드} growing / declining"
  찾는 것: 수요 트렌드, 성장/쇠퇴 신호, 관련 투자 동향

Agent 3 — Technical Precedent (Builder 모드 또는 기술 아이디어)
  검색: "{핵심 키워드} github"
  검색: "{핵심 키워드} open source implementation"
  찾는 것: 기존 구현, 참조 아키텍처, 선례의 함정
```

검색 실패 시: 실패를 명시적으로 기록하고 계속 진행.

## Phase 3: Premise Challenge (전제 도전)

인터뷰 + Landscape 결과를 바탕으로 핵심 전제 1~2개에 도전한다.

```
"이 아이디어의 핵심 전제는 {전제}입니다.
 그런데 {반대 증거}가 시사하는 바가 있습니다.
 이걸 어떻게 보시나요?"
```

**Anti-sycophancy 규칙**:
- "재밌는 아이디어네요" — 금지
- "좋은 방향이에요" — 금지
- 증거 없이 긍정 — 금지
- 대신: 입장을 취하고, 반론을 제시하고, 사용자의 응답을 기다린다

## Phase 4: 판단 문서

**판정 기준**:

| 판정 | 기준 |
|------|------|
| **GO** | 명확한 수요 증거 + 경쟁 공백 확인 + 실행 가능한 최소 쐐기 존재 |
| **PIVOT** | 아이디어의 핵심은 유효하나 방향 수정 필요. 구체적 피벗 방향 제시 |
| **NO-GO** | 수요 증거 부재 / 이미 포화 시장 / 실행 가능성 부재 중 2개 이상 |

**판정은 입장이다** — "판단하기 어렵습니다"는 없다. 증거에 기반한 명확한 입장을 취한다.

---

판단 문서를 `.ina/specs/{YYYYMMDD-HHMM}-explore-{slug}.md`에 저장:

```markdown
# Explore: {아이디어 제목}

**판정: GO / NO-GO / PIVOT**
**모드**: Startup / Builder
**날짜**: {날짜}

## 핵심 발견

- {발견 1}
- {발견 2}
- {발견 3}

## 경쟁 Landscape

{Landscape 요약 + 소스 인용}

## 핵심 전제와 도전

**전제**: {전제}
**도전**: {반론 + 증거}
**사용자 응답**: {사용자 입장}

## 판정 근거

{GO/NO-GO/PIVOT을 선택한 구체적 이유}

## 다음 단계

{판정에 따른 구체적 행동 제안}
```

## 실행 브릿지

판단 문서 저장 후:

- **GO** → `/ina:think`로 스펙 구체화 제안 (추천)
- **PIVOT** → 피벗 방향을 새 입력으로 `/ina:explore` 재실행 제안
- **NO-GO** → 판단 존중. "이 판단을 번복하고 싶다면 `/ina:explore`를 다시 실행하세요."

## 입출력

- **입력**: 자연어 아이디어 설명
- **출력**: `.ina/specs/{YYYYMMDD-HHMM}-explore-{slug}.md` (GO / NO-GO / PIVOT 판정 문서)
